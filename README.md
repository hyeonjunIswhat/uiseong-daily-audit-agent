# 효규가영(効規可寧) — 의성군 일상감사 검토 AI 에이전트

> 효율적으로 · 규정에 맞게 · 편안하게.
> 계약을 집행하기 **전에** 감사부서가 미리 검토하는 절차(일상감사)를 돕는 멀티 에이전트입니다.
> **의성군청 폐쇄망 Open WebUI에 실배포되어 운영 중**이며, 이 저장소는 그 코드 전체입니다(실물 감사 문서·키 제외).

## 무엇을 하나

공무원이 의뢰서·공고문·산출내역서(hwp / hwpx / xlsx / pdf)를 **그냥 첨부하면**:

1. **1차 — 즉시 사전점검 (LLM 0회, 5초 이내)**
   파일별 원문 확인 → 문서유형·사업유형 분류 → 일상감사 **대상 판정**(의성군 규정 별표 실기준) →
   산출내역 xlsx **산식 자동 검산** → 문서 간 금액·사업명 **교차 대조** → 필수서류 완결성 확인 + 보완요청 문안 생성
2. **2차 — 핵심 AI 점검 (LLM 정확히 1회)**
   규정 §7④ 8축 루브릭 전 항목을 **단일 구조화 호출**로 검토 (qwen3:14b, 로컬 Ollama)
3. **3차 — 감사팀용 의견서 초안 (지적이 있을 때만 +2회)**
   지적후보 전체 **일괄 문맥검증 1회** → IRAC 의견서 합성 1회. 지적이 없으면 둘 다 생략(결정론 템플릿)

결과 화면 최상단은 결론부터: **진행 가능 / 보완 후 진행 권고 / 담당자 확인 필요** + 먼저 조치할 사항(파일명·근거·다음 조치).

## 오늘 작업 — 성능 회차 (2026-07-15)

"문서 1건에 10분"을 최우선 성능 결함으로 규정하고 호출 그래프를 구조부터 바꿨습니다.

| 경로 | 변경 전 호출 수 | 변경 후 호출 수 | 변경 전 시간 | 변경 후 시간(실측·실물 문서) |
|---|---:|---:|---:|---:|
| 자가점검 | 4 (트리아지 1 + 축별 3, 본문 중복 전송) | **1** | 84~300초 | **70.2초** |
| 지적 없는 서면검토 | 6 | **1** | 수 분 | **12.2초** (KV 캐시 히트 시) |
| 지적 있는 서면검토 | 6+ (지적당 검증 1콜씩) | **3** | 10분+ | 예산 최대 300초 |

핵심 변경:

- **축 선별에서 LLM 제거** — 8b 트리아지 1콜을 규칙 신호 사전(`rules/axis_signals.yaml`)으로 교체.
  좁히기 전용 · 신호 부족 시 전축 폴백 · 스킵 축은 사유와 함께 표시(침묵 금지).
- **축별 순차 호출 폐지** — 활성 축 전체를 항목 나열 1콜로(`AxisReviewer.review_all`).
  축 식별자·항목 추적성은 회신을 재그룹해 유지. 스키마에 `minItems`를 동적으로 강제해
  모델이 일부 항목만 답하고 멈추는 현상을 해결(실측: **판정 미회신 25건 → 0건**).
- **검증 일괄 1콜** — 지적후보 전체를 한 프롬프트로(`ContextVerifier.check_all`),
  동일 법령 조문은 중복 없이 1회만 수록. 미회신 항목은 지적 유지(검증은 강등 전용).
- **입력 다이제스트** — 8,000자 초과 문서는 금액·산식·표제·축신호 줄만 결정론 발췌(`rules/digest.py`),
  `[문서: 이름]` 출처 마커 보존, 생략 시 "원문 N자 중 M자 검토" 명시.
- **타임아웃·예산 규율** — 용도별 HTTP 타임아웃(검토 90 / 검증 45 / 합성 90초, 요청 단계에서 실제로 끊김),
  타임아웃 재시도 금지(부분 결과 우선), 시간 예산 자가점검 120초 / 서면 300초 —
  초과 시 완료된 결과를 즉시 반환하고 남은 항목만 `확인 필요`.
- **호출 계측** — 모든 LLM 호출이 `stage·model·request_id·입출력 크기·소요·상태`를 perf 로그에 남김
  (문서 원문·프롬프트는 코드 수준에서 아예 전달하지 않음).
- **호출 예산을 테스트로 강제** — `tests/test_perf_budget.py`가 자가점검 1콜 / 서면 무지적 1콜 /
  서면 지적 3콜을 초과하면 즉시 실패. 성능 검증은 전부 모의 지연(실제 모델 속도 비의존).

같은 날 UX 회차: 파일을 읽는 즉시 실제 근거를 스트리밍(`📄 [원문 확인] \`요청서.hwp\` — 본문 554자`),
진행 메시지 최대 간격 15초(추상 문구·점 펄스 금지), 1차 결과 선반환(이후 실패해도 유지),
취소 시 늦은 이벤트 폐기, 개인정보(주민번호·전화·이메일·계좌) 화면 마스킹.

## 아키텍처 — 결정론 우선 멀티 에이전트

```
사용자 (Open WebUI 말풍선)
   │  파일 첨부 / 본문 붙여넣기 / 자연어 질문
   ▼
IntentClassifier (규칙, 10종 의도)          ← LLM 0회
   ▼
┌─ 1차 즉시 사전점검 (전부 결정론) ──────────────────┐
│ 파서(hwp·hwpx·xlsx·pdf, 무의존 자체 구현)          │
│ BusinessClassifier → 별표 룰엔진 대상판정          │
│ 산식 검산 · 문서 간 교차 대조 · 서류 완결성        │
└──────────────────────────────────────────────┘
   ▼
규칙 트리아지(축 신호 선별) → 통합 검토 1콜 (qwen3:14b)
   ▼ (지적 있을 때만)
1차 결정론 검증(조문 실존·인용 태그) → 일괄 문맥검증 1콜 → 의견서 합성 1콜
```

- **감독형 + 고정 DAG** (스타 토폴로지). 협업형(에이전트 간 토론)은 재현성 때문에 의도적으로 배제.
- **멀티 에이전트 = 역할·검증 책임의 분리**이지, 에이전트마다 모델 호출을 따로 한다는 뜻이 아님.
- 검증은 **강등 전용**(flag-only): 2차 검증은 지적을 '확인 필요'로 내릴 수만 있고, 새 지적을 만들거나 결정론 판정을 번복하지 못함.
- 판정·금액·기한 등 숫자가 걸린 것은 전부 룰엔진(동일 입력 → 동일 결과). LLM은 서술 판단에만.

## 운영 환경

- **폐쇄망 로컬 전용**: Ollama(qwen3:14b / 8b) + Open WebUI 0.9.x in-app Pipe Function. 외부 API로 문서가 나가지 않음.
- 배포는 `batch/deploy.sh` 하나로: 코드 반입 → Function DB 갱신 → 배포 스탬프(장수 프로세스 모듈 핫리로드) → 스모크.
- 운영 로그는 `code_only`: 의도·모드·길이·소요·오류만. **문서 원문과 개인정보는 로그에 저장하지 않음.**
- 법령 근거는 국가법령정보 API로 실존 검증된 조문만 인용(클릭 → 원문 칩). 기준금액은 의성군 일상감사 규정 별표(ELIS 원문 자동 확보) 기반.

## 테스트

```bash
python -m unittest discover -s tests    # 232건 — LLM·네트워크 불필요(전부 모의 주입)
```

파서 단위부터 호출 예산 회귀까지 전부 포함. 성능 테스트는 모의 지연 클라이언트로 호출 수와 예산을 검증합니다.

## 이 저장소에 없는 것

실제 감사 문서·골든셋(실물 사건), API 키·.env, 운영 로그는 **의도적으로 제외**했습니다.
루브릭 항목·기준금액은 감사팀 협의 전 잠정값(provisional)이며 모든 산출물에 그 고지가 강제 삽입됩니다.
결과는 AI 초안이고, 최종 판단은 언제나 담당자와 감사인의 몫입니다.
